# 知到 ZhiDao · 字体排印系统（Typography）
## —— 一个品牌、两种载体（浅色 HTML 文档 ↔ 深色 PPT）的统一规格

> **本册角色**：《知到》设计系统 · 字体排印分册（上级文档：[../design-system.md](../design-system.md)）
> **事实来源**：`docs/prd/zhidao-prd-v1.2.html`（web 侧唯一事实源）与 `docs/share/__back/《从0到1，用AI Agent搭一个能跑的产品》.pptx`（41 页，deck 侧唯一事实源；引用中的 slideN.xml 行号指其解包后的 `ppt/slides/`）。
> **校验方式**：逐断言溯源核查（font-size / letter-spacing / line-height / text-transform 全量计数；sz / spcPct / typeface / spc 属性全片枚举；关键页逐 run 解析 + 渲染图抽查），2026-07-14。
> 单位约定：web 用 px / em；deck 用 pt（XML `sz`÷100）与 EMU（914400 EMU = 1in）。

---

## 0. 结论先行（五句话版）

1. **两载体的中文正文其实是同一款字**：deck 的「思源黑体」= Source Han Sans = web 栈里的「Noto Sans SC」（Adobe/Google 同源双发行名）。这是整个双载体统一的**锚点**，不是巧合，应写进规范固定下来。
2. **三层字族分工是身份，具体字体是可替换域**：`展示层（几何 grotesk）/ 内容层（现代中文黑体）/ 元信息层（等宽 mono）` 的三层语法不可动；层内换字（Space Grotesk→别的 grotesk、Noto→MiSans）不伤品牌。
3. **字号在 11.5–31 的「阅读区间」两边数字孪生**（web px ≈ deck pt 同数值同职责：正文 16px↔15–16.5pt、页题 24–31px↔29pt、元信息 11–12.5px↔11.5–12.5pt）；只在两端分叉——web 顶端用 clamp 响应式放大（至 76/80px），deck 为投影插入 web 没有的「金句层 26–28pt」和「演示宣告 34pt」。
4. **mono 标签语法是全系统最强的身份特征**：`等宽 + 10–13.5px + 大写 + .04–.22em 宽字距 + 语义色`，凡编号/路径/状态/图例一律走它；deck 目前用「青绿小字粗体 + 手工空格」降级模拟（XML 已核实全片 0 处 `spc=` 字距属性），迁移时应补齐。
5. **字重两载体折叠映射**：web 四档（400 正文 / 500 mono 标签 / 600 标题 / 700 正文强调），deck 只有 b=0/b=1 两档——deck 的 b=1 吞并 web 的 600+700，层次差由颜色（#E6EDF3/#8A97A3/#16C79A/#E0B84A）接管。

## 1. 双载体字体栈与映射

### 1.1 现状栈（精确值）

**Web（HTML L9、L20–22）**，Google Fonts 引入 Space Grotesk 400/500/600/700 + JetBrains Mono 400/500/600 + Noto Sans SC 400/500/700：

```css
--disp: "Space Grotesk","PingFang SC","Noto Sans SC","Hiragino Sans GB","Microsoft YaHei",sans-serif;
--body: "PingFang SC","Noto Sans SC","Hiragino Sans GB","Microsoft YaHei",-apple-system,"Segoe UI",Roboto,sans-serif;
--mono: "JetBrains Mono",ui-monospace,SFMono-Regular,Menlo,monospace;
```

**Deck（slideN.xml `<a:rPr>` 逐 run 设置，无母版承载）**：
- 主导：`思源黑体`（全片 2192 处 typeface 属性，latin/ea 槽各 803 个 run 显式设为它——如 slide10.xml L117–119 三连；其中约 586/803 个 run 是 latin/ea/cs 三槽全设，其余 run 的 cs 槽落在雅黑）。主导地位属实，但「绝大多数 run 三槽全设」是主导倾向而非全称——见下条反例；
- 混入：`微软雅黑` 共 180 处属性（latin 12 / ea 12 / cs 156）。cs 槽兜底**系统性散布 34 页**（156 处，cs 只管复杂文种，通常不影响中文字形）；真正影响字形的 latin/ea 级混入共 12 处，集中在 **4 页**——S2 正文 14pt、S35 **页题 29pt + 暗行 22pt**、S36 页题 29pt + 正文 16pt、S38 论述 21pt。注意：全片 9 个含「▶」的符号 run（S3/9/16/22/27/34 章封面 19pt + S20/25/30 34pt）三槽**全是思源黑体**，并未混入雅黑；
- 特例：`Roboto` 共 3 处，均为 `<a:cs>` 槽（slide1.xml L288、slide2.xml L129/L233，两页），只影响拉丁/复杂文种兜底；`+mn-ea` 主题字散见 **11 个文件共 32 处**（S2×8、S3×2、S9×1、S16×1、S22×2、S27×1、S34×2、S35×3、S36×4、S38×1、S41×7）——theme1 是 Office 默认（latin=Calibri、ea 为空串），意味着这些 run 的实际字形**不受控**。

### 1.2 映射关系与配置逻辑

| 语义层 | Web | Deck 现状 | 为什么这样配 |
|---|---|---|---|
| 展示层（标题/品牌） | Space Grotesk 600（中文回退黑体） | 思源黑体 b=1 | web 中英混排标题里英文呈几何 grotesk 味=品牌记忆点；deck 全中文语境无拉丁展示需求，靠 29–30pt+粗体立骨架 |
| 内容层（正文） | PingFang→**Noto Sans SC**→雅黑 | **思源黑体**（=Noto Sans SC） | 两边实为同一字形家族；Windows 环境 web 命中 Noto Sans SC webfont，与 deck 完全同源 |
| 元信息层（标签/编号/代码） | JetBrains Mono 500–600 | **缺席**——用思源黑体小字号+青绿色+手工空格模拟 | deck 生成链（python-pptx）未引入 mono；身份在此处有断层 |
| 拉丁兜底 | Roboto（--body 尾部） | Roboto（S1/S2 两页 cs 槽 3 处）/ 思源黑体拉丁 | 巧合重合，均非身份 |

### 1.3 身份 vs 可替换域

**身份（换掉即不像「知到」）**：
1. 三层字族语法本身（展示/内容/元信息各配一族，元信息必须等宽或视觉上「等宽感」）；
2. 中文黑体系无衬线基调（全系统 0 衬线字、0 楷宋）；
3. 展示层「600 字重 + -.01em 负字距」的立骨方式（不靠 700 加黑）；
4. 思源黑体/Noto Sans SC 这一对同源字（两载体同一张中文脸）。

**可替换域（层内自由换）**：
- Space Grotesk → 任何几何人文 grotesk（Cabinet/Sohne/Manrope 等），保留 600/-.01em 即可；
- JetBrains Mono → IBM Plex Mono / Fira Code 等，保留大写+宽字距语法即可；
- PingFang/Hiragino 等平台回退项可按目标平台增删；
- deck 的微软雅黑兜底应**删除而非替换**（统一锁思源黑体）；Roboto 两页 3 处均可清理。

**待核**：deck「思源黑体」具体是哪个地区版/字重版（XML 只写「思源黑体」，未见 CN/SC/TC 后缀与 Medium/Bold 独立字重名）；渲染机是否安装、未安装时 PowerPoint 落到哪款字——v1.1 起生成侧由 `assert_fonts_installed()` 显式检查兜住（缺字报错而非静默替换）。

## 3. 「mono 标签语法」完整规格

### 3.1 语法公式

```
mono 标签 = 等宽字族 + 10–13.5px + [uppercase] + letter-spacing(.04–.22em) + 字重 500/600 + 语义色
```

**铁律**：凡**编号、路径、状态、协议、图例、字段名**一律走此语法（元信息层）；内容层（body）与它严格分离——SVG 图内的节点主标签 `dg-lbl` 用 body 而非 mono，就是「图里的正文仍是正文」的范例。

### 3.2 Web 侧构件全表（精确值，源 CSS 已逐条复核）

| 构件 | 字号 | 字距 | 大写 | 字重 | 颜色 | 职责 |
|---|---|---|---|---|---|---|
| `.zd-eyebrow` | 12px | **.22em**（最大档） | ✓ | 400 | amber `#C8932E` | hero 眉题 |
| `.zd-nav-title` | 11px | .14em | ✓ | — | mute `#8A8C82` | 侧栏「目录 · Contents」 |
| `.zd-metastrip dt` | 11px | .1em | ✓ | — | mute | 档案条字段名 |
| `.zd-kicker` 章号 | 13px | — | — | 600 | clay `#B5532F` | 章号徽章 01–13 |
| `.hx` 小节号 | 12px | — | — | 500 | moss `#3E7A5C` | 1.1 / A.2 |
| `.zd-chip` | 12px | — | — | 500 | 四色变体 | 行内语义胶囊 |
| `.zd-call .ic` | 11px | .06em | ✓ | 600 | pine/amber/clay | 引注条徽章（定义/铁律/病根） |

| `.zd-mini .num` | 12px | — | — | 600 | clay | `PRINCIPLE 01` 卡片眉 |

| `.zd-pill` | 13.5px | — | — | 500 | `#E6E4DC` | hero 论点胶囊 |

配套规律：**字号越小、语义越标签化 → 字距越大**（10–11px 配 .07–.22em；12–13px 配 0–.06em）；正文永远 0 字距；大标题 -.01em。text-transform 全文档共 **8 处声明**（全部 uppercase）：eyebrow（L45）、metastrip dt（L61）、nav-title（L69）、clh（L81）、node-tit（L82）、call .ic（L120）、th（L137）、us-ac span（L203）。

### 3.3 Deck 上对应什么（跨载体迁移）

**已核实的硬事实**：全 41 页 XML **无一处 `spc=` 字符间距属性**——deck 的「宽字距」全部是手打空格（「第 一 章 · 共六章」逐字加空格）；也没有等宽字族。现状是一套**降级模拟**：

| Web mono 构件 | Deck 现状对应 | 现状规格 |
|---|---|---|
| eyebrow 眉题 | S1 场合行 / 章封面 kicker「接下来」 | 14pt 绿 B / 16pt 灰 |
| kicker 章号 + nav 编号 | 右上面包屑「章 · 节」+「第 X 章 · 共六章」 | 章 12.5pt `#16C79A` B · 节 11.5pt `#8A97A3`；进度语 14pt 绿 + 手工空格 |
| chip 语义胶囊 | pill 胶囊（烤进 PNG）/ 表内 ✓✗ emoji | 位图，不可改 |
| mini .num 编号眉 | 01–04 圆徽 / 编号牌（烤进 PNG） | 位图 |
| th 表头 | 原生表头（反白：`#16C79A` 底 + `#0C1418` 深字，15pt B） | 亮底深字替代「小字大写」 |
| code / .fn 文件名 | 终端窗内 `DESIGN.md`（19pt `#16C79A` B）+ 红黄绿点窗件 | 终端窗组件承担「代码语境」 |
| soulbadge 状态胶囊 | ▶ 当前项指针 + 品牌绿描边高亮框（原片实测 1.2/1.3pt 混用；v1.1 定版收口 1.2pt=STROKE_BRAND） | 形状语法替代文字语法 |
| legend 图例 | 图内烤字 | 位图 |

**迁移规范（沉淀为规则）**：
- **A 路线（新 deck 推荐）**：为编号/路径/状态/公式引入 JetBrains Mono（仅拉丁数字场景，中文标签仍思源黑体）；python-pptx 里对这类 run 单独设 latin=JetBrains Mono。风险：目标放映机字体缺失 → 需嵌入字体或转 PDF【待核：放映环境】；生成机侧可把字体名一并交给 `tokens_deck.assert_fonts_installed()` 显式检查（v1.1 新增）。
- **B 路线（兼容现状）**：保持思源黑体，用「11.5–14pt + `#16C79A`/`#8A97A3` 双色 + 粗体 + 手工空格」四通道模拟；手工空格仅限 ≤6 字的导航短语（长句加空格会破坏换行与检索）。
- 不论哪条路线，**元信息层的三个身份必须保住**：小半档字号（比正文小 2–4 级）、语义色（绿=结构/当前，灰=次级）、「·」分隔语法。

## 4. 中英混排规则

### 4.1 「·」间隔号（两载体共用的第一混排符）

五种用法，全部两侧留空：

| 用法 | Web 例 | Deck 例 |
|---|---|---|
| ① 中英对照粘合 | 「灵魂 · CORE」「进 · INPUT」「目录 · Contents」 | —（deck 少用中英对照） |
| ② 并列枚举 | 「纯本地 · 零运维 · 不上 RAG」 | 「本次分享 PPT · 知到 PRD v1.2 · …」（S41，绿条目白间隔点） |
| ③ 编号与名称 | 「US-01 · 入库链接」「FLOW 01 · 问答流」 | 「第 X 章 · 共六章」 |
| ④ 导航层级 | —（web 用 § 交叉引用） | 面包屑「章名  ·  (N) 节名」（**双空格** `"  ·  "`，且可扩到三段，S40） |
| ⑤ 口号串联 | 「编译 > 检索 · 往回织 · 用得出」（footer） | 「清晰沟通·小步快跑·持续验证·降低成本」（图内） |

细节差：web 单空格、deck 面包屑双空格——迁移时统一为「中文语境两侧各一个空格；deck 面包屑允许双空格换呼吸」。

### 4.2 中英对照标题（cn/en 双行）

- Web 规格：`.cn` 中文大字 `clamp(40px,6.8vw,80px)`（600、1.02 行高、-.01em）+ `.en` 英文副题 `clamp(18px,2.4vw,26px)`（`.06em` 宽距、`#7E8479` 灰绿）——**中文极短立名、英文长句作定义**；词汇表 dt = 中文 700 + `small` mono 英文对照。
- Deck：**无 cn/en 双行体例**（全中文演讲语境），英文只作术语行内混排（Agentic Coding、MVP、AI Native），字体跟随思源黑体拉丁字形；S1/S2 例外见 §1.1 Roboto 条。迁移判断：cn/en 双行是**文档载体专属**，不必硬搬进 deck；deck 需要英文点缀时，用「灰旧词 → 绿新词」行内递进（S28）代替。

### 4.3 数字与代码

- Web：一切数字编号（nav 计数器、ol 序号、01–13 章号、US-01、P0/P1/P2、v1.0→v1.2）与代码/路径/字段名一律 mono；表内 `code` 12.5px 灰底 5px 圆角。
- Deck：数字用思源黑体粗体+颜色扛（23pt 章号、19pt ▶ 行）；公式直接排原生文字——S25 公式本体「score = 2×标签重叠 + 1×标题重叠 ≥ 4 → 双向关联」为 **17pt #16C79A 粗体**，其上引导行「不上向量库：」为 **19pt #E6EDF3 常规**。迁移：deck 引入 mono 后，公式与文件名优先换 mono。

### 4.4 letter-spacing 档位总表（web 专用；deck 以空格模拟）

| 档位 | 用在哪 | 规则 |
|---|---|---|
| **-.01em** | h1、h2 大标题 | 展示字负距收骨架 |
| 0 | 一切正文、卡内说明、中文长句 | 正文永远不加距 |
| .04em | topmeta（11.5px） | 最小标签档 |
| .05em | 表头 th（11.5px）、ingest-arrow | 表格元信息 |
| .06em | clh、node-tit（L82）、call .ic（11–12px）、h1 .en | 徽章与英文副题 |
| .07em | us-ac span（10px） | 验收眉 |
| .08em | soulbadge（10.5px，L165）、road-tag（L235）——全文档仅此 2 处 | 胶囊徽章 |
| .1em | metastrip dt、dtitle span（11px） | 档案字段名 |

## 5. 行高 / 字重体系

### 5.1 字重职责（web 四档 → deck 两档折叠）

| 字重 | Web 职责 | Deck 映射 |
|---|---|---|
| 400 | body 正文、eyebrow、注解 | b=0 |
| 500 | mono 标签基础档（chip/pill/hx） | b=0 + 语义色（色替重） |
| 600 | **展示层唯一标题字重**（brand/h1/h2/h3/dtitle）+ mono 强标签（kicker/ic/num/th）+ SVG dg-lbl | b=1 |
| 700 | 只给正文内 strong、h4、lname、gloss dt、dg-lbl-core | b=1 |

规律原文：「展示字靠 600 + 负字距立骨架，强调靠 700 只给正文」。deck 因思源黑体单字重文件 + b 开关，把 600/700 折叠为 b=1，层次缺口由**颜色四档**补——`#8A97A3` 压灰 → `#E6EDF3` 常规 → `#16C79A` 结构强调 → `#E0B84A` 金句强调（S35 三行「亮→暗→金」即「颜色当字重用」的范例）。

### 5.2 行高梯度

**Web（unitless）**：h1 `1.02` → h2 `1.15` → nav `1.35` → 卡内说明 `1.5–1.6` → td `1.55` → lede `1.65` → **body `1.72`** → code pre `1.7` → footer meta `1.9`。极值实域为 1–1.9：另有 `.zd-step .sn{line-height:1}`（L216，单字符大数字的排版归零特例）与 hero lede 一处内联 `1.8`。
**Deck（spcPct÷100000，已从 XML 全量核出）**：表格 `1.10`（S12/14/18/24/32 统一 110000，另 S1×2）→ S1/S3 `1.10–1.15` → 段落 `1.18–1.35`（S26 118/130/135、S28 120/130、S10 120/125）→ 终端窗 `1.25`（S20/25/30）→ `1.50`（S2×1、**S33×5、S36×2**，共 3 页）→ `2.00`（S19×5、**S33×1**，装饰性撑高）。1.5/2.0 仍属少数特例，但不止 S2/S19 两页。未设 spcPct 的 run 为默认 100%。

### 5.3 场景规则（1.72 vs 1.5x 的分界）

| 场景 | 行高 | 判据 |
|---|---|---|
| 连续多段阅读（web 版心 ≤1160px，一行 35–45 字） | **1.72** | ≥2 段连排必用；这是「编辑部感」的呼吸底盘 |
| 卡片/说明内 ≤3 行短文 | 1.5–1.6 | 容器自带 padding 呼吸，行高收紧防松散 |
| 表格单元格 | web 1.55 / deck 1.10 | deck 表格行高由等高行（0.7–0.9in）承担呼吸，行内压到 1.1 |
| 大标题 | 1.02–1.15 | 字号越大行高越紧（两载体同规律） |
| 金句/终端宣告（deck） | 1.25 | 一两行短句，站得住即可 |
| mono 元信息行 | web 最松（footer meta 1.9、code 1.7） | 「元信息行高最松」是 web 侧刻意反差 |

## 7. 两载体差异总对照

| 维度 | Web（浅色 PRD） | Deck（深色 PPT） | 统一策略 |
|---|---|---|---|
| 字族数量 | 3 族（disp/body/mono） | 1 族（思源黑体）+2 兜底 | deck 补 mono 或用四通道模拟；中文已天然同源 |

| 大写标签 | text-transform 8 处（全 uppercase）+内容大写 | 无（中文语境） | 大写仅限拉丁标签，中文靠「小字号+色」达意 |

| 数字编号 | mono 全托管 | 思源黑体粗体+色 | deck 有 mono 后优先切换 |

## 8. Do / Don't

### Do
1. **元信息与内容分族**：编号、路径、状态、图例、字段名一律 mono（web）或「小 2–4 级字号 + `#16C79A`/`#8A97A3` + 粗体」（deck），绝不混进正文字族层级。
2. **标题统一 600 + -.01em**（deck 为 b=1），一页只有一个 29–31 级标题；金句在 deck 提到 26–28pt 但**永远低于页标题**。
3. **mono 大写标签按「字号↓则字距↑」配距**：10–11px 用 .07–.22em，12–13px 用 0–.06em；eyebrow 独享 .22em。
4. **「·」作万能粘合符**：中英对照、编号-名称、并列枚举、导航层级都用它，两侧留空格。
5. **中文正文两载体锁同一家族**：web 栈保证 Noto Sans SC 可命中（webfont 已引入），deck 每个 run 的 latin/ea/cs 三槽都显式设思源黑体。
6. **行高按场景取值**：连排长文 1.72；卡内短文 1.5–1.6；表格 web 1.55 / deck 1.10；标题 1.02–1.15。
7. **数字孪生互译**：跨载体定字号时先查 §2.3 表，11.5–31 区间 px↔pt 同数字直接换单位。
8. **可改文案永远原生字**：标题、导航、金句、结论条、表格必须是 HTML 文本 / PPT 原生 TextBox·tbl，方便改稿与检索。

### Don't
1. **别给中文长句/正文加 letter-spacing**——正文档位是 0；宽字距只属于 ≤13.5px 的拉丁标签；deck 的手工空格只限 ≤6 字导航短语。
2. **别用 700 当标题主力**（浅纸底上会发闷）；700 只给正文 strong、h4 与 SVG 核心标签。
3. **别把 mono 用于中文正文或图内主标签**——SVG 里 `dg-lbl` 用 body 字族是既定范例；mono 承载中文会破字面节奏。
4. **别在 deck 混入微软雅黑或放任主题字**（S2/S35/S36/S38 四页 latin/ea 级混入雅黑、cs 槽雅黑散布 34 页、11 文件 32 处 `+mn-ea` 已是既成瑕疵）；生成脚本必须逐 run 锁字体。
5. **别把可改文字烤进图**——37/39 页整页文字入位图导致不可检索、缩放发虚、改稿重生成，是本 deck 最大排印教训。
6. **别让金句/大字越过标题字号天花板**：deck 金句上限 28pt（<29 页题）；40.5/44 只属于封面与谢幕两页。
7. **别新造字号**：两边梯度已覆盖 10–80px / 11.5–44pt，共 17 级固定值（+5 条 clamp）+ 27 级，新需求先向现有级取整。
8. **别把 web 的响应式 clamp 逻辑搬进 deck**（deck 画布固定 13.33×7.5in），也别把 deck 的金句字级搬回 web（web 用 call 组件表达金句）。
