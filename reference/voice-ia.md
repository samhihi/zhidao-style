## 结论(先行)

这套品牌的文案系统只有**一条声线、两套体例**:

1. **一条声线**:第一人称亲历(「我重砸第二个」)+ 判断先行(verdict 可以是「不值得收」)+ 反空话(每句口号都能指到一个机制:「编译 > 检索」指向 relate.py 的打分公式)+ 承诺—兑现闭环(S4 许诺「散场前兑现」→ S41 逐项交付;PRD 路线图 → 「✓ 实证注记」)。
2. **两套体例**:文档走「编辑部档案体」——双位编号 01–13、mono 元信息层全大写英文、changelog 显式留痕、§ 交叉引用;演讲走「现场剧场体」——中文序数双轨(一、+(一))、面包屑三段式、每页一论点、结论条收口。**编号系统是两载体最硬的分界:文档用阿拉伯双位与 §,演讲用中文序数;绝不互串。**
3. **句式是品牌的最小可迁移资产**:对比句(A,不是 B)、箭头句(→)、大于号句(>)、「·」间隔号并列、括号翻译层、破折号补刀——六种句式在两载体逐字复用,是「同一个人在说话」的直接证据。
4. 命名系统三轴:`zd-` 前缀弱化 BEM(变体用单词类不用 `--modifier`)、`data-l` 语义轴驱动色彩叙事、「日期-标题」文件名 + 「vX → vY」修订节点。

以下逐节给可复用骨架。**精确值均出自 XML/CSS 原文;推断处标「待核」。**

## 1. 文档体例(HTML · 编辑部档案体)

### 1.1 章节编号双轨语法:kicker 01–13 + hx + h4

三层标题、两套编号、一次对齐:

```html
<section class="zd-sec" id="overview">                     <!-- 锚点用英文语义词 -->
  <div class="zd-sec-head">
    <span class="zd-kicker">01</span><h2>产品概述与设计哲学</h2>
  </div>
  <p class="sub">用一页讲清:知到是什么、为谁而做、和市面上「AI 知识库」的根本不同在哪。</p>
  <h3><span class="hx">1.1</span>一句话定义</h3>
  <h4>小节内再分点时才用 h4</h4>
</section>
```

| 层级 | 编号 | 字体/字号/颜色(精确值) | 写法规则 |
|---|---|---|---|
| 章 h2 | `zd-kicker` 手写 `01`–`13` | mono 13px,clay `#B5532F` 字 + `#D3CDBC` 描边徽章,7px 圆角;h2 disp 600 `clamp(24px,3vw,31px)` `-.01em` | 章题=名词短语,可带冒号分层:「核心功能:进 / 存 / 取」 |
| 章引 `.sub` | 无 | 14.5px graphite `#565851`,`margin-left:56px` 与章号对齐(≤560px 取消) | 一句「这章干什么」,常以动词开头:「用一页讲清:」「一句话锁死位置,……」 |
| 小节 h3 | `.hx` `1.1`–`12.3`,附录改 `A.1`–`A.3` | hx 为 moss `#3E7A5C` mono 12px;h3 18.5px disp 600 | 小节题=动宾或判断短语:「验收标准:你自己用 14 天会不会扔」 |
| 段题 h4 | 无编号 | 15px 700 | 只在小节内需要第三层时出现,不进目录 |

- 侧栏目录用 CSS 计数器 `counter(nv, decimal-leading-zero)` 自动生成 01–13,**与 kicker 手写号一一对应**——双轨必须同步,加章先改两处。
- 交叉引用体例:正文括注 `(§4.2 / A.2)`;changelog 条目尾缀 `(§N)`。引用是双向的:US-01/02 ↔ §9 矩阵 P0 行互指。
- 13 章的宏观信息架构本身可复用:概述/哲学 → 问题 → 定位/用户/故事 → 功能 → 架构 → 选型 → 数据 → 实现方法 → 范围/优先级/路线 → NFR → 风险 → 验收/度量/开放问题 → 附录(术语+规则+命名)。

### 1.3 用户故事卡 + 验收口径

```html
<div class="zd-story">
  <div class="us-id">
    <span>US-01 · 入库链接</span>              <!-- clay mono:编号 · 短名(3–5 字) -->
    <span class="fl">FLOW · 进</span>          <!-- pine mono 流向标:FLOW · 进/取,CORE · 编译 -->
  </div>
  <div class="us-body"><b>作为</b>…,<b>我想</b>…,<b>以便</b>…。</div>
  <div class="us-ac">                          <!-- 虚线上沿分隔 -->
    <span>验收</span>给一个链接 → 生成一篇含「摘要 + 3–5 要点 + 3–5 标签 + verdict」的
    note,并更新 INDEX.md。                    <!-- 字段名用 <code> -->
  </div>
</div>
```

验收口径的三条句法(全部来自 US-01~06 原文):
1. **输入 → 产出**:「给一个链接 → 生成一篇含…的 note,并更新 INDEX.md」;
2. **流程链**:「AI 先查索引 → 读相关结论页 → 给综合答案并附来源」;
3. **允许/而非**(否定性口径,最见立场):「允许出现『不值得收』」「附来源,**而非**甩关键词列表」。

编号体系:`US-01`–`US-06`,名字 3–5 字、以四字为主(入库链接 / 立场摘要 / 想法发散 / 跨资料问答[5 字] / 主动派活 / 往回织[3 字]);与 §9 优先级矩阵 P0/P1/P2 交叉引用。

### 2.6 41 页章节节奏与密度曲线

**总构成:41 = 开场 2 + 六章(1 封面 + N 正文)+ 致谢 1;备注口径「正片 35 页」= 41 − 6 张章封面。**

| 章 | 页码 | 封面+正文 | 正文页型序列 | 密度走势 |
|---|---|---|---|---|
| 开场 | S1–S2 | 0+2 | 全图扉页(原生文本 4 组:kicker + 主标题 + 副题 + 署名)→ 路线目录(6+1 条) | 极低 → 中 |
| 一 为什么自己搭 | S3–S8 | 1+5 | 四卡受众 → 金句呼吸 → 引用背书 → VS → VS+结论条 | 中高→极低→中→高→高 |
| 二 产品思维 | S9–S15 | 1+6 | 互动指令 → 信息版画(全片密度天花板)→ 判据表格 → VS → 六格表 → 带走双卡(烤图) | 极低→极高→中→中高→中→中 |
| 三 架构思维 | S16–S21 | 1+5 | 三栏流程 → 表格实证 → 前后截图铁证 → **演示** → 四坑 | 高→中高→高→极低→中高 |
| 四 技术思维 | S22–S26 | 1+4 | VS → 四字诀表 → **演示** → 结论+双卡 | 高→中→极低→中高 |
| 五 Agentic Coding | S27–S33 | 1+6 | 概念递进+金句卡 → 四步流程 → **演示** → 合规声明 → 分工表 → 最小样本双卡 | 中低→中高→极低→中→中高→中 |
| 六 结尾 | S34–S40 | 1+6 | 回收开场 → 金句框 → 三场信息图 → 大字论述+结论条 → 三卡 CTA → 心里话交棒 | 低→低→高→低→中→极低 |
| 尾 | S41 | 0+1 | 谢幕+资料领取 | 低(但每项都是钩子) |

节奏规律(可直接迁移):
1. **每章内部锯齿**:封面(低)→ 方法论核心(高)→ 演示/金句(极低,呼吸)→ 带走(中)。高密度页后必接呼吸页(S11 之后是 S12 表格、S10 之前;S5/S10/S20/S25/S30/S35 全是刻意呼吸位)。
2. **演示恒定落位**:三、四、五章各 1 场,全部在本章第(三)节——「概念讲两节,第三节动真格」的节拍器。
3. **章页数 4–6 页**,节数 4–6 节;一节最多 2 页(S13/14、S18/19),超 2 页拆节。
4. **第六章反转**:无演示、无表格,密度阶梯下降、情绪阶梯上升(24 白 → 22 灰 → 28 金粗的字号-情绪同构,S35),以「全场最该慢」收束。
5. 双受众带走页出现在**二、四、五章末**(讲给两个工种的章);纯认知章(一、三、六)不发糖,改金句或铁证收尾。

## 3. 文案风格配方(两载体逐字通用)

### 3.1 对比句「A,不是 B」——立场的标准句法

| 实例 | 出处 | 场景 |
|---|---|---|
| 「这些工具是**仓库,不是助理**」 | PRD §2(zd-call「病根」条) | 给旧事物定性;S7 以 VS 句式复现同一对立(标题「Agent ≠ 聊天框:仓库 vs 助理」) |
| 「MVP 是**滑板车,不是半辆车**」 | PRD §9 / S13 标题 | 借喻立判据 |
| 「AI 时代真正稀缺的,**不是执行力,是定义力**」 | S1 副题/S36 金句 | 全场母题 |
| 「**当采购员,不当工程师**」 | S23 标题 | 角色定位 |
| 「**不是 Vibe,是 Agentic** Coding」 | S28 标题 | 概念辨析 |
| 「不是聊天——**编排了一个 Agent,跑完了一条流水线**」 | S33 | 成果重述 |

适用:标题、金句、定性判断。变体「机械归脚本,语义归 Claude」(归字对仗)。**规则:B 位必须放真实存在的竞品/误解,不树稻草人。**

### 3.2 箭头句「X → Y」——流程与跃迁

| 实例 | 出处 | 场景 |
|---|---|---|
| 「答得出 → 用得出」 | PRD hero / footer | 价值跃迁(两个三字动补) |
| 「读原文 → 提炼 → 打标签 → 算关联 → 存文件 → 往回织」 | PRD 词汇表 ingest 条 | 工序链(动宾×6) |
| 「喂上下文 → 分步推进 → 每步验证 → 卡住止损」 | PRD §8 / S29 | 方法四步 |
| 「(图纸 → 已建成)」 | changelog 实证卡标题 | 状态跃迁,嵌括号 |
| 「AI 起初稿,我只审、只改 → DESIGN.md」 | S20 终端窗 | 动作→交付物 |

适用:验收口径、流程页、版本注记。**规则:箭头链每节点 2–5 字动宾;链长 ≤6;终点必须是名词交付物或状态。**

### 3.3 大于号句「A > B」——排序型断言

| 实例 | 出处 |
|---|---|
| 「编译 > 检索」 | 全品牌第一口号(PRD pill/SVG/footer + S6 标题 + 章小注,两载体≥7 处复现) |
| 「文件 > 数据库」 | PRD §6 选型 |
| 「Python > TS/JS > 其他」 | PRD §6(可三连) |

适用:选型立场、方法论排序。**规则:> 两侧是同类可比名词;一个产品只留一句一级大于号口号(此处是「编译 > 检索」),其余降级到选型节。**

### 3.4 带立场短句——判断落锤

- 「可以写『不值得收』」(PRD ingest 规则——把否定权写进流程)
- 「贪多是第一杀手」(PRD §11 风险)
- 「简单,就是最大的稳定」(S26 标题)
- 「沉没成本不值钱,跑通才值钱」(PRD §8)
- 「痒点是产品坟场的建材」(S11 备注)
- 「讲得清的土办法,胜过讲不清的黑箱」(S26 前端卡)

适用:call 结论条、卡片收尾、备注自我提醒。**规则:≤14 字,主语可省,必须可反驳(有人会不同意才叫立场)。**

### 3.5 「·」间隔号——并列的品牌标点

| 用法 | 实例 |
|---|---|
| 三连立场 | 「纯本地 · 零运维 · 不上 RAG」「分步派活 · 每步验证 · 闭环执行」 |
| 中英对照位 | 「目录 · Contents」「灵魂 · CORE」「进 · INPUT」 |
| 编号 · 名称 | 「US-01 · 入库链接」「FLOW · 进」「第 一 章 · 共六章」 |
| 双关键词节名 | 「验证 · 带走」「三问 · 先画『取』」「简单即稳定 · 边界」「几句心里话 · 交棒」(S40) |
| 清单流式分隔 | S41 资料 5 项、S26「按渠道抓取 + 纯 Markdown + …」(并列升级用 +) |
| 日期档案 | 「✓ 实证注记 · 2026-06-28」「初稿 06-21 · 最近更新 06-28」 |

**规则:「·」只做并列与对照,不做转折;并列 2–5 项;PPT 面包屑里的分隔 run 用两个半角空格夹「·」(U+00B7)。同级并列要动作感时换「+」(知到 = A + B + C + D)。**

### 3.6 中英混排——什么时候上英文

| 上英文 | 实例 | 不上英文 |
|---|---|---|
| 元信息层(mono,全大写) | `PRINCIPLE 01` `FLOW · 进` `CORE` `RAW` `PRD v1.2` `AI NATIVE KNOWLEDGE BASE` | 判断句、金句永远纯中文 |
| 无公认汉译的术语 | Agentic Coding / AI Native / MVP / RAG / frontmatter / ingest | 有汉译的一律中文(编译,不写 compile) |
| 文件名/代码/字段 | `CLAUDE.md` `DESIGN.md` `relate.py` `INDEX.md` `verdict` `raw/` | |
| 品牌双语(中文在前) | 「知到 ZhiDao」「知到 · ZhiDao」 | |
| 术语首现的对照小字 | 词汇表 `<small>compile</small>`、`vs Vibe Coding` | |

**总规则:英文=协议层(名字、路径、状态),中文=判断层(立场、情绪、结论)。**PPT 英文密度显著低于文档:41 页原生文本里,英文只出现在不可译术语与品牌名(Agentic Coding、AI Agent、MVP、Vibe Coding、GitHub、Claude Code、Claude/GPT)、文件名与公式;文档里 mono 元信息层几乎全英文——因为文档有「档案感」需求,演讲要「说人话」。

### 3.7 括号补充——语气的第二声道

| 括号装什么 | 实例 |
|---|---|
| 翻译(抽象→具体) | 「不是执行力(写代码),是定义力(想清楚)」「要什么(产品)· 怎么流(架构)· 用什么(技术)· 怎么指挥(Agentic)」 |
| 一句话定义 | 「Vibe Coding(一句话变 App)」「MVP(v1)」 |
| 口语补刀 | 「打 3–5 个标签(不够就少打,别硬凑)」「『知到』GitHub 项目(clone 即用)」 |
| 状态跃迁 | 「✓ 实证注记 · 2026-06-28(图纸 → 已建成)」 |
| 范围限定 | 「这是『个人库』量级的结论」(S26,自限边界) |

**规则:括号只装翻译、定义、补刀三类,不装第二论点;一句话最多一层括号。**近亲是破折号补刀「——」:装转折升华(「你没有贬值——是你的稀缺点搬了家」「知到不用——把『理解』前置成编译」),破折号比括号语气重,一页最多一次。

### 3.8 其他登记在案的句式

- **四零排比**:「零数据库、零向量库、零服务器、零部署——简单就是最大的稳定」(排比后必须接破折号落锤);
- **张力对仗规则观**:「写太死它就僵,写太松它就飘」「想清楚了又快又好,想不清楚越快越糟」;
- **三把刀口诀**(疑问三连):「还成立?能不要?不做会死人?→ 砍」;
- **交棒句**(对偶祈使):「给流程以智能,给 AI 以意图。」(S40,整句 28pt B 金——整句金色是全 deck 的最高礼遇句式,另一处同礼遇是 S35 尾句「我想清楚了一件值得做的事。」同为 28pt B #E0B84A)。注意 deck 内金色实有 `#E0B84A` 与 `#FFC000` 两个 hex(FFC000 可见于 S36/S40 关键词;S2 处仅为被全幅图盖住的隐藏页底遗留,渲染不可见)——已按 GOLD(暖金)/GOLD_HOT(亮金,仅限「临门一脚」)两档分工收编进 tokens_deck.py,迁移时照分工取用(见 §7)。

## 4. 命名系统

### 4.1 CSS:`zd-` 前缀 + BEM 弱化

- **组件一律 `zd-` 前缀**:`.zd-card` `.zd-call` `.zd-chip` `.zd-story` `.zd-gloss` `.zd-changelog` `.zd-kicker`…(HTML 内 `.zd-` 前缀类共 64 个,含子类与布局类;组件级类全部带前缀,无一例外);
- **弱化 BEM**:不用 `__element` / `--modifier`;子元素用**后代短类**(`.zd-changelog .clh` `.node-tit`;`.zd-story .us-id .fl`),变体用**单词态度类**:`.tight`(密度)、`.pine/.amber/.clay/.ink`(语义色)、`.pos/.neg`、`.bad/.good`、`.yes/.no`、`.ok/.stop`、`.now`、`.live`——**变体名直接是判断词,类名本身带立场**;
- 布局工具类数字化:`.zd-g2/.zd-g3`(栅格)、`.zd-loop4`(四步)。

### 4.2 `data-l` 语义属性——用数据轴驱动色彩叙事

```html
<div class="zd-layer" data-l="raw">…</div>    <!-- clay  #B5532F 原料/只读/警戒 -->
<div class="zd-layer" data-l="notes">…</div>  <!-- moss  #3E7A5C 半成品/过程 -->
<div class="zd-layer" data-l="wiki">…</div>   <!-- pine  #1E5A44 核心/结论(+渐变底+「灵魂 · CORE」徽章) -->
<div class="zd-layer" data-l="briefs">…</div> <!-- amber #C8932E 产出/回环(文字用深档 --amber-text;原片 #946a10 已降考古,v1.1 定版 #7D5A0D) -->
```

这条映射同步出现在 SVG 节点(`dg-box-raw` / `dg-box-note` / `dg-core`)、图例、优先级徽章(P0=clay、P1=moss)——**clay=入口与警示、moss=过程、pine=核心结论、amber=回环与产出,是横贯组件/图表/文案的同一条叙事轴**。迁移法:新项目先定 3–4 个语义档位词(如 raw/notes/wiki/briefs),再让 `data-l` 吃掉所有变体着色,不给每个组件单独发明色类。

### 4.3 文件命名

| 类型 | 体例 | 实例 |
|---|---|---|
| 留底/笔记/选题 | `日期-标题.md`(YYYY-MM-DD 或短日期前缀) | `notes/2026-06-28-xxx.md`、`briefs/「日期-选题.md」` |
| 编译页 | `topic-<主题>.md` / `entity-<名>.md` | `wiki/topic-agentic-coding.md` |
| 交付文档 | `<产品>-<类型>-v<版本>.<ext>` | `zhidao-prd-v1.2.html` |
| 技术方案 | `<主题>-<日期紧凑>.md` | `telegram-bot-local-20260706.md` |
| HTML 锚点 | 英文语义单词 | `#overview #problem #position … #appendix` |
| 需求编号 | `US-0N` + 3–5 字短名(四字为主);优先级 `P0/P1/P2` | `US-06 · 往回织` |

### 4.4 版本体例(v1.0 → v1.1 → v1.2)

- 版本号只用两段 `vX.Y`;**修订记录按「节点」组织**:每个 `node-tit` 是一次跃迁 `vX → vY`(斜体 mono),下挂动词开头条目;
- 大版本语义:v1.0 初稿 → v1.1 补方法与验收(结构补全)→ v1.2 实战返修(与实现对齐);
- **「修订」与「实证」分卡**:计划性变更进琥珀卡,落地验证进松绿卡——文档同时保留「图纸态」与「建成态」两种时间线;
- 顶栏与 metastrip 同步露出版本(`PRD v1.2` / `初稿 06-21 · 最近更新 06-28`),**版本出现处 ≥3(顶栏/metastrip/footer),改版本要改三处**——机械活,适合脚本化。
- PPT 侧版本体现在资料包命名(「知到 PRD v1.2」)与备注引用(「zhidao-prd-v1.1.html」拉上大屏)——**对外引用永远带版本号**。

## 5. 两载体语气差异与共同声线

### 5.1 差异对照表

| 维度 | A · 文档(编辑部严谨) | B · 演讲(口语断言) |
|---|---|---|
| 章节编号 | 阿拉伯双位 01–13(mono 徽章)+ 小节 1.1/A.1 + § 交叉引用 | 中文序数双轨:章「一、~六、」+ 节「(一)~(六)」,无 § |
| 导航形态 | 侧栏 13 条目录 + 粘性进度条 + IntersectionObserver 高亮 | 面包屑「章绿节灰」+ 章封面进度菜单 + 「第 X 章 · 共六章」 |
| 句子长度 | 长句+从句+括注,一段 2–4 句 | 一页 ≤3 句,一句 ≤2 行,金句独占一页 |
| 立场载体 | call 徽章词(定义/判据/铁律/病根/真相/诚实边界) | 颜色与字号(金句金 26–28pt、行动绿、灰句压 22pt) |
| 英文密度 | 高:mono 元信息层全大写英文 | 极低:只留不可译术语、品牌名与文件名 |
| 人称 | 「我(作者)/你(读者)」,用户故事用「作为…我想…」 | 「我(现场)/各位/大家」,还有点名工种「给产品同事」 |
| 证据形态 | 表格 ×14、代码块 ×6、SVG 架构图 + § 互引 | 现场演示 ×3、真截图铁证(S19)、「实打实走一遍」 |
| 修订痕迹 | changelog 双卡显式留痕,矛盾显式标注 | 页面零留痕;备注存讲稿指针(讲稿 §N 行 X–Y)与 🔴 风险旗标 |
| 结论位置 | 章内 call 条随论点走 | 页底结论条(灰引导语 + 绿粗结论)或金句框,「对比页必须给结论」 |
| 受众分流 | US-01~06 按流向(FLOW · 进/取/CORE) | 「给产品同事(绿)/给前端同事(金)」双卡按工种 |
| 幽默额度 | 低,只在括号补刀(「别硬凑」) | 中,允许「骂娘」「自嗨」「发糖」级口语 |
| 承诺管理 | 路线图 → 实证注记核销 | 开场承诺(S4)→ 散场核销(S41)+ 备注「务必兑现」 |

### 5.2 共同声线(一个人格的五条不变量)

1. **第一人称亲历**:「我重砸第二个」(S21)/「我只审、只改」(S20)/「你自己用 14 天会不会扔」(PRD §12.1)——证据永远从自己身上取。
2. **判断先行且可否定**:摘要必带 verdict,允许「不值得收」;开放问题给「推荐/开放」状态;S26 主动说「但得说清边界」。
3. **反空话**:口号必须有机制兜底——「编译 > 检索」落到 relate.py 公式,「往回织」落到 S19 前后截图,「零运维」落到目录树。写不出机制的口号删掉。
4. **承诺—兑现闭环**:凡「散场前兑现」「图纸 → 已建成」类句子,必有对应的核销位(S41 清单 / 实证注记卡)。
5. **给规则留度**:「写太死它就僵,写太松它就飘」——规则文案自身要示范弹性(「3–5 个,不够就少打」),不写绝对化频度词(必须/每篇/一律,除非真的一律)。

## 6. 迁移到新项目的最小集

新项目(新产品文档 + 新分享)只需带走 12 件:

1. 前缀命名:`<brand>-` 组件前缀 + 单词态度变体类 + `data-l` 语义轴(先定 3–4 个档位词);
2. 文件名 `日期-标题.md` / `<产品>-<类型>-v<X.Y>.<ext>`,对外引用带版本号;
3. 双轨编号纪律:文档 = 双位阿拉伯 + §,演讲 = 中文序数 + 括号序数,互不越界;
4. metastrip 四槽:是什么 / 第几版 / 管多宽 / 活没活(状态槽独享主色);
5. changelog 双卡:琥珀修订(动词开头 + §尾注)+ 松绿实证(日期 + 状态跃迁括号);
6. 用户故事三段卡:`US-0N · 3–5 字短名(四字为主)` + 流向标 + 作为/我想/以便 + 「输入 → 产出」验收(允许否定口径);

## 8. Do / Don't

### Do

1. **每个口号配一个机制**:写「编译 > 检索」前,先确认能指到 relate.py/往回织截图这类实物;指不到就别喊。
2. **状态与颜色绑死再写文案**:修订=琥珀、实证=松绿、行动=青绿/松绿、金句与警示=金;写文案时先问「这句话是什么状态」,再落到对应容器。
3. **模板句逐字复用,只换变量**:章封面备注、面包屑、「本章 · 要讲这几节」、「给 X 同事」卡——固定成分一字不改,变量只有章名/节数/受众名。
4. **验收口径写成「输入 → 产出」**,产出带字段清单与 `<code>` 字段名,并保留一条否定性口径(「而非甩关键词列表」)。
5. **承诺登记**:开场/开篇许下的每件事(带走 5 件东西、散场前兑现)入表,结尾逐项核销,一项都不许溜。
6. **英文只上协议层**:文件名、编号、状态、无汉译术语;判断句、金句、情绪词永远中文。
7. **高密度页后排呼吸页**:连续两页 20+ 信息点是事故;金句页/演示页/互动页就是节拍器。
8. **目录即摘要**:每章(每节)标题后跟一句立场小注,让扫读者只看目录也能带走论点。
9. **敢写自限**:「诚实边界」「个人库量级的结论」「不值得收」——每份材料至少一处主动划界,可信度靠这个挣。

### Don't

1. **别让同一口号长出两个版本**:「定义力,才稀缺」vs「想清楚,才稀缺」是现成教训;口号入库,复用时只准引用不准重写。
2. **别跨载体串编号**:演讲里不出现 §、01–13;文档里不出现「(一)(二)」「第 X 章 · 共六章」。
3. **别把要维护的文案烤进图**:S37/S39 文字入位图后不可检索、改稿要重生成;凡是会改的字必须是活字(原生文本/表格)。
4. **别硬凑满格式**:标签「3–5 个,不够就少打」;规则写死(「每篇必打 5 标签」)AI 就硬凑,人也一样。
5. **别用含糊词收清单尾**:「等」「相关」「一些」判死;清单必须点得清数量(「5 件东西」「四个坑」「三判据」)。
6. **别一页多爆点**:金色强调每页 ≤1 处(PPT),破折号补刀每页 ≤1 次,call 条不连排;爆点密了等于没有爆点。
7. **别树稻草人**:对比句 B 位必须是真实竞品或真实误解(收藏夹/Vibe Coding/半辆车),不虚构敌人。
8. **别让摘要没立场**:verdict 字段不许空、不许「中性客观」;写不出判断就写「不值得收」,这也是判断。
