# 知到设计系统 v1.0 对抗性评审报告

> **修复记录（2026-07-15）**：29 条发现已全部修复并升版 v1.1（应用户要求，P2「记债不修」项一并清偿）。功能实测（add_veil 渲染半透明验证、页眉对照原片 S18）与逐条对账双验证通过。逐项落点见 `design-system.md`「九、修订记录」。

## 评审方式

四个人格并行独立评审：资深视觉/品牌设计师（15 年，主导过两个大厂设计系统）、资深 UX/交互设计师（可用性与无障碍背景）、资深产品经理（设计系统治理与 DevEx 背景）、挑刺评测用户（明天要照 SKILL.md 干活的执行者）。评审对象为 docs/design/ 全部资产（总纲 design-system.md、tokens.css、tokens_deck.py、五分册 reference/*、zhidao-style 技能），共产出 44 条发现。随后逐条回源核实：解包原 41 页 pptx 逐 slide XML 比对、python 重算 WCAG 对比度、实跑 python-pptx 1.0.2 复现 API 限制、全库 grep 验证「唯一提及」类断言、渲染图目检。核实结果：**39 条 CONFIRMED、5 条 PARTLY（失实局部已附修正表述）、0 条 REFUTED**；去重合并后 **29 条**。

## 总评

四位评审对「考古质量」高度一致：逐值溯源、例外如实标注、对比度实算、跨文件核心 hex 零漂移，作为审计档案接近满分，实测口径经得起对原片 XML 的独立复核——这在个人项目的反向沉淀里是罕见水位。分歧不在事实而在定性与后果排序：**视觉评审**认为要害是「中间件缺位」——审计做完了、生产件没交付（组件 CSS、页题坐标、章封面配方），它更像审计报告而非生产系统；**UX 评审**判定它本质是「静态出版物外观规范」而非设计系统，交互/无障碍/动效三层完全空白比视觉缺口更根本，且对比度豁免逻辑把 WCAG 用反；**PM 评审**聚焦治理——七入口矛盾在交付当天已存在、派生物零同步机制、deck 产线有不可执行步骤，但同时明确 semver/评审会对一人+AI 属过度治理，最低配即可；**挑刺执行者**实跑后把唯一 blocker 钉在压暗渐变纱上：python-pptx 公开 API 根本做不出带 alpha 的三停纱，「所有数值可放心复用」的承诺在最后一公里失守。共同结论：这套系统**当档案读可信，当作业指导书用会翻车**；44 条发现无一被驳回，问题密度真实存在，但绝大多数修法是小时级的。

## 发现

### Blocker

#### 1. 压暗渐变纱（三层三明治第二层）在规定工具链下无实现路径，照做会产出盖死插画的错误制品
- **提出人**：挑刺执行者（UX 从「命门未做成可执行件」角度同报）
- **核实结论**：CONFIRMED
- **证据**：SKILL.md 任务二第 2 步要求「压暗渐变纱（双色 BG_ABYSS/BG_DEEP，90° 为主，停靠点按画面亮度调）」；color.md §5.3 称其为「可读性的命门」，债#7 模板为三停带 alpha（顶 100%／中 pos≈69% α≈25%／底 84%）。实跑 python-pptx 1.0.2：`fill.gradient()` 仅生成 2 停，`gradient_stops[2]` 抛 IndexError，GradientStop 无任何 alpha/transparency 属性（核实补充：新版有 gradient_stops 位置/颜色 API，但停靠点透明度仍不支持，模板需 α≈25%，必须手写 `<a:gradFill>` OOXML——7 份文档无一提及）。tokens_deck.py 只有 GRADIENT_ANGLE_DOWN/UP 两个角度常量加注释，无任何构造函数。按公开 API 直做并渲染，得到一块不透明蓝黑幕布，完全遮住下层插画。
- **为什么**：这是每页必有的层（motifs §3.4「B 的 41 页页页有全幅底图」+纱），规格写得极细却零可执行代码；明天照做的 agent 要么静默产出错误视觉（不透明纱=插画白生成），要么自己发明 XML 手术——两者都违背「所有数值可放心复用」的承诺。债#7 只沉淀了参数模板，不解决实现缺口。
- **建议修法**：tokens_deck.py 增加 `add_veil(slide, stops=[(BG_ABYSS,0,100),(BG_DEEP,69,25),(BG_ABYSS,100,84)], ang=GRADIENT_ANGLE_DOWN)` 助手函数，内部做 gradFill XML 注入（含 alpha）；SKILL.md 任务二第 2 步改为「调用 add_veil，默认后半场模板停靠点」。

### Major

#### 2. deck 骨架不可复现：最高频骨架件缺纵向锚点常量、无 helper 函数、「最大技术债」（母版空白）未入债表
- **提出人**：视觉 + 挑刺（几何常量）、UX + PM（可执行件与母版债）——四人各自撞上同一断层
- **核实结论**：CONFIRMED
- **证据**：tokens_deck.py 页题只有 `TITLE_X = Emu(777240)`；实测 slide18.xml 页题文本框 @(777240, 475488) ext(10607040, 868680)——y/cx/cy 三值在 7 份规范文档中零命中（layout-components §1.5 只写「标题 @x=777240，29pt」），y=475488 为 33 个 29pt 页题中 23 页的主档。同类缺失：HEADER_BAR_Y_CHAPTER（slide3 实测 502920）、CHAP_BOX_X/Y（658368, 1627632）、TABLE_Y、终端三点起点（S20 @2697480, 2459736）、QUOTE_W/H 有尺寸无坐标。页眉三件套只有 9 个裸 EMU 常量，全文件唯一函数是 set_font()；layout-components §6.D 自认「41 页全部逐页手放，是最大技术债，骨架应进 slideLayout 母版」（41 页 rels 全指 slideLayout7 blank，解包复核属实），但 design-system.md 第八节债表 7 条无此条，SKILL 任务二仍教逐页手放。附带：债5「同一 deck 内固定一句」没锁定哪句（voice-ia §7 已实证建议「定义力，才稀缺」）；债6 定版 2048×1152 未回应 layout-components §5.3 自提的 4K 待核。
- **为什么**：页眉三件套是「骨架恒定」第一原则的头号载体，纵向锚点全靠猜——拿 HEADER_BAR_Y 猜 TITLE_Y 偏差 91440 EMU=0.1in，每个 agent 各猜一版，恒定骨架从第一页就漂移；想做准就得回原片解包 XML，那设计系统就没完成它的工作。39 页 × 每页 3 个手放形状，改一次页眉规格要动 39 处，正是债表要防的事被工具链固化成了标准作业。
- **建议修法**：补齐 TITLE_Y=475488、TITLE_W/H、HEADER_BAR_Y_CHAPTER、CHAP_BOX_X/Y、TABLE_Y、底部安全线（6.9in）等常量——核实补充：页题框带 spAutoFit，S24/32/36/37 四页 y/cy 随行数变化，真正恒定的是垂直中心 909828 EMU，宜同时沉淀 TITLE_CY_CENTER=909828 并注明 y=475488 为双行档，而非把 y/h 当无条件恒量；加 `add_header(slide, title, chapter, section)`、`add_conclusion_bar()` 函数把逐页手放收敛为单点维护；母版债补进第八节；债5 锁「定义力，才稀缺」，债6 补「4K 放映升 3840×2160」。

#### 3. deck 产线第 2 步「全幅 AI 插画」无任何落地工具，也无降级路径，流水线卡死
- **提出人**：PM、挑刺
- **核实结论**：CONFIRMED
- **证据**：SKILL.md 任务二 step2「全幅 AI 插画（用 motifs 分册 §3.2/3.3 提示词模板生成，2048×1152，暗部 ≥60%，文字压暗区）」；design-system.md 迁移指南 deck step2 同；motifs §3.4「41 页页页有全幅底图」写成每页必备层。实测 scripts/ 全量清单与技能清单无任何生图工具；全库 grep 降级/退化/纯色/fallback，SKILL、总纲（含迁移指南与债表）、motifs、tokens_deck.py 均无「无生图工具时怎么办」。
- **为什么**：没看过原 deck 的 agent 走到 step2 只能卡住或擅自发挥——而「自行发挥」正是设计系统要消灭的东西；QA 项「检查文字是否压在暗区」同时失去对象。HTML 产线每步可执行，deck 产线假装断点不存在。
- **建议修法**：SKILL.md 第 2 步写明降级链：①有生图工具→按提示词模板生成；②用户提供成图→校验 2048×1152/暗部比例；③都没有→退化为 BG_DEEP 纯底 + 压暗渐变纱 + 四角光斑（注明这是可接受的合规知到风，L13/表格页本就只需四角透光），QA 项相应放宽。

#### 4. L2 章封面（每个 deck 必用 ×6）的唯一规格是「文字烤进 PNG」，与自家三条禁令和生图模板互斥，新 deck 无合规产出路径
- **提出人**：视觉
- **核实结论**：CONFIRMED
- **证据**：layout-components L2 行「右侧 ~4.8–5.4in 见方『本章要讲这几节』烘焙卡」、voice-ia §2.2「整卡烤在 PNG 里…卡内 (一)–(N) 清单」；与三处禁令直接互斥——layout-components Don't#1、motifs Don't#4、voice-ia Don't#3「可维护文案别烤进位图」，且 motifs §3.2/3.3 生图提示词模板写死「禁止出现文字、水印」。L9（8 页）至少进了设计债#3，L2 连债都没记。deck 原生示意图也无规格：motifs §4 深底示意图几何全是 HTML 侧 dg-* SVG 值，§2 图标「线宽为估值」且无矢量资产；tokens_deck.py 有章封面高亮框常量却无议程卡原生几何，而 SKILL 与迁移指南又锁死「14 原型内选、不发明新版式」。
- **为什么**：agent 做新 deck，到第 3 页（第一张章封面）就必须二选一：照 L2 规格把章节清单烤进 AI 生图（违反禁令且生图模型排不出可靠中文），或自创原生排版（违反「不发明新版式」）。最高频仪式页无合规配方，产出必然各次不同。
- **建议修法**：为 L2 议程卡补一版原生实现规格（装饰底图 + 原生文字清单 + 图标用字符/简单形状）；给 deck 原生流程图/示意图一页画法（节点色已有，补 roundRect+箭头的几何与线宽）；「烘焙」明确降级为纯氛围层专用。

#### 5. tokens.css 语义层深面泄漏：--warm-text/--brand-2 两深面未覆写、--border-2 在 deep 面语义翻转、deep 面 --negative 装了自家规则禁止的值
- **提出人**：PM、挑刺
- **核实结论**：CONFIRMED
- **证据**：:root 有 --warm-text:#946A10、--brand-2:#3E7A5C，但 [data-mode="ink"]（L71–88）与 [data-mode="deep"]（L94–111）均未覆写→深底继承浅面值。实算：#946A10 on #181A1F = 3.59:1、#3E7A5C = 3.43:1，均不达 AA，违反 color.md Do#5「深底上的字先过 4.5:1 再上屏」——而 color.md §2.1 明明给了深面对位（沙金 #C9A86A/#E0B84A，brand-2 的 ink 对位雾绿 #BFE3CF、deep 无对位）。--border-2 三面 #D3CDBC/#34383F/#16C79A：deep 面从中性强线翻成品牌强调边。deep --negative:#FF5F56 与 color.md §6 提名的 --zd-neg-dark:#E2A4A4（双底 AAA）矛盾，且 §2.3 明确 #FF5F56 为「组件专属，非语义色」。文件尾部「跨面禁忌」注释与债表均未覆盖。
- **为什么**：SKILL 和迁移指南都要求「组件样式只引用语义层」——恰恰是守规矩的组件会踩坑：var(--warm-text) 进 hero/代码窗静默降级成烂对比度；var(--border-2) 在 deep 面全部变绿边，绿边滥发=「看这里」语法失效；var(--negative) 产出系统自己禁止的深面红色负向。语义层契约破洞比裸写 hex 更危险，因为它披着「合规」外衣。
- **建议修法**：ink 面补 --warm-text:#C9A86A、deep 面补 --warm-text:#E0B84A；--brand-2 两深面给对位色或显式注释「深面禁用」并指回 --brand；--border-2 拆成 --border-strong（永远中性）与 --border-accent（品牌强调），deep 的 #16C79A 挂到后者；deep --negative 改 #E2A4A4，终端红拆成 --dot-red 组件色。

#### 6. 对比度数值错抄/精选/张冠李戴：「所有数值可放心复用」的承诺在最高频入口（tokens.css、SKILL 硬规则）失守
- **提出人**：视觉、UX（两条）、挑刺
- **核实结论**：CONFIRMED
- **证据**（三组，均经 WCAG 公式独立重算）：① tokens.css L30「--amber-text: #946A10…(4.64:1)★」是对卡片底 surface #FBFAF5 的精选值，对默认页底 paper #F5F3EC 实算 4.37:1 AA✗——color.md §4.1 自己记着「4.37 | AA ✗」，附注 5 还给了修复值「深化至 ~#7D5A0D」（实算 5.66 达标）；而迁移指南与 SKILL QA 无背景限定地强制「琥珀文字必用深档 #946A10」，债表只收 mute/clay 未收 amber-text。② tokens.css L116 与 SKILL 硬规则「#8A97A3 放纸底只有 1.96:1」——实算 2.69:1；1.96 是 #16C79A 上纸底的值（color.md §4.4 归属正确），确系抄串行的张冠李戴。③ 琥珀 #C8932E「2.62:1」是对 surface，paper 实测 2.47（color.md 正确），双基准均未注明。结构性原因（UX）：每个数值平均出现 3–5 处、无权威所在册机制，七入口在 v1.0 发布当天就没对齐——改一处漏四处是结构必然。
- **为什么**：tokens.css 与 SKILL 速记是 agent 引用频率最高的入口，★ 号明示「安全」，而 --amber-text 的主用途场景（页底上的 11–13px mono）恰恰不满足 4.5:1；错误数字长在入口位置会被优先执行与传播，任何复核者对整套「逐断言校验」的可信度打折。
- **建议修法**：短期：全系统声明统一对比基准（建议 paper #F5F3EC，最不利面）；--amber-text 定为 #7D5A0D 或双值如实并列注明；1.96→2.69（顺手把 1.96 标回 #16C79A）、2.62→2.47，tokens.css/SKILL/color.md 三处同步。长期：给每个重复数值指定权威所在册（对比度=color.md、几何=token 文件），其他入口只引用不复述数字，SKILL 硬规则只写结论（「不达标」）不写数。

#### 7. --mute 的对比度豁免逻辑把 WCAG 用反：3.07:1 的弱化色被限定在最需要 4.5:1 的 10–12px 小字上使用
- **提出人**：UX
- **核实结论**：CONFIRMED
- **证据**：tokens.css L33「--mute: #8A8C82; /* 弱化文字(3.07:1, 只用于 11-12px 元信息, 不承载正文) */」；design-system.md 债#4 定版同款（L141 迁移指南再强化）；color.md §4.1「只能用于 ≥18px 或装饰性元信息」。WCAG 2.1 SC 1.4.3 大字豁免线是 24px regular / 18.66px bold（3:1），18px regular 不豁免，且例外仅 incidental/装饰/logotype，无「短元信息」豁免；PRD 实际用例 11/11/10px（L61/L69/L203）全部远低于豁免线。实算 3.07:1 精确吻合。color.md L187 自己也承认 deck 弱化灰 6.76:1「比 HTML 的 mute 健康得多」。
- **为什么**：小字需要更高对比而非更低——规则倒置后写进了 token 注释和设计债定版，agent 照此生产的每份 HTML 都带批量 AA 失败的 10–12px 文字，且自认为合规；低视力读者在最需要读清「验收/字段名」这类元信息时读不清。
- **建议修法**：二选一并改三处（tokens.css 注释、债#4、color.md §4.1）：a) 把 --mute 加深到真正 ≥4.5:1——核实注：评审原建议的 #767871 在 paper 底仅 4.03:1 仍不达标，需加深到约 #6E706A（4.52:1）；b) 保留现值但改规则为「仅限纯装饰性、移除后不损失信息的元素」，必读元信息一律 --graphite。同时删掉「≥18px 可用」的错误豁免线，改为「≥24px regular / ≥18.66px bold」。

#### 8. web 侧只交付了色彩 token：约 30 个 zd-* 组件的全部几何散落在分册散文表格里，组件 CSS 零交付
- **提出人**：视觉
- **核实结论**：CONFIRMED
- **证据**：tokens.css 除色彩外仅 --maxw/--r/三条字族（L38–44）；组件规格——call「15px 18px 内距+3px 左条+10px 圆角」（layout-components L151）、card「22px 24px」（L133）、7 级响应式断点（§5.1 L260）、12 档圆角刻度——全部只存在于 markdown 表格（圆角刻度在 tokens.css L39 有一行注释，但注释非 token）；间距刻度仅一句建议「统一时建议以 4px…为原子步长」（§1.1 L49）未成 token；SKILL 任务一只说「把 tokens.css 拷进目标项目」，未指向任何组件实现；债表 7 条无一覆盖此断层。
- **为什么**：「新 HTML 文档直接引入」的可复现性主张实际只覆盖色板；组件层每次由 agent 从散文重新翻译成 CSS，内距/圆角/断点的漂移是必然，且无从 diff 校验——这是「答得出」与「用得出」之间最大的断层。
- **建议修法**：从 zhidao-prd-v1.2.html 抽出经校验的 zd-components.css（组件几何+断点+print）与 tokens.css 并列交付；或最低成本：SKILL 明写「组件 CSS 一律从 PRD 源码复制，分册表格仅作核对」。间距/圆角刻度进 token（--sp-1…--sp-n、--r-chip/--r-card/--r-canvas/--r-pill）。

#### 9. 整套「可复用」web 规范零交互状态：无任何 hover/focus/active/disabled/error 的 token 或规则，原件唯一的 hover 实践在沉淀中失传
- **提出人**：UX

#### 27. 字体兜底不闭环 + set_font() 顺序缺陷：FONT_FALLBACK 是死常量（缺字体时 PowerPoint 静默替换），对已带超链接的 run 盲目 append 会产出 schema 非法文件
- **提出人**：PM（字体检查）、挑刺（hlink 顺序 + 死常量）
- **核实结论**：CONFIRMED
- **证据**：tokens_deck L38–39 FONT="思源黑体"、FONT_FALLBACK="微软雅黑"（注释「正式产出前确认渲染机装了思源黑体」）——FONT_FALLBACK 定义后全仓库零引用，set_font() 只用 font 参数；typography 附待核#1 自认「渲染机是否安装、未安装落到哪款」未知，L59 已建议删除雅黑兜底但无代码机制。实测复现：对先设 hlinkClick 的 run 调 set_font()，rPr 子元素序变为 [solidFill, latin, hlinkClick, ea, cs]——ea/cs 落在 hlinkClick 之后，违反 CT_TextCharacterProperties 顺序约束（PowerPoint 可能弹修复；原片零 hlinkClick，属前瞻场景）。
